# Copilot 提交信息生成规范

> **版本**: v0.2.0.8 | **日期**: 2026-04-20 | **作者**: 开发团队 | **更新**: Copilot集成、成杯提交规范

---

本文件作为 Copilot 生成 Git 提交信息的指令，所有生成的提交信息必须严格遵循以下规范。

---

## 零、文件来源约束

body 中列出的每一个文件路径，**必须来自上下文中实际存在的 diff 内容**，禁止：

- 编造任何未出现在 diff 中的文件路径
- 推测或描述 diff 中未体现的功能变更
- 将多个文件的变更合并为一条（除非规则四第 5、6 条明确允许）

---

## 一、整体结构

```
type(作用域): 简短中文主题

### 模块名
- `相对文件路径`：做了什么 → 产生的影响

BREAKING CHANGE: 描述破坏性变更（仅在有不向下兼容的变更时写）
Closes #N（仅在关联 issue 时写）
```

- **首行**（header）：不超过 70 个字符，不加句号
- **空行**：header 与 body 之间必须有一个空行；body 与 footer 之间必须有一个空行
- **body**：按模块目录分组，每个文件单独一条，使用反引号标注文件路径
- **footer**：仅在有破坏性变更或关联 issue 时写，否则省略

---

## 二、提交类型（type）

| 类型       | 用途                                                         |
| -------- | ---------------------------------------------------------- |
| feat     | 新增功能                                                       |
| fix      | 修复 Bug                                                     |
| refactor | 代码重构（既不是新增功能，也不是修复 Bug）                                    |
| perf     | 性能优化                                                       |
| style    | 代码格式调整（不影响逻辑，如空格、缩进、命名规范）                                  |
| test     | 新增或修改测试代码                                                  |
| docs     | 文档变更（仅限 `help_document/`、`README.md` 等说明文件）               |
| build    | 构建系统或外部依赖变更（如 `requirements.txt`、打包脚本）                     |
| chore    | 其他不影响源代码或测试的杂项变更（如 `.gitignore`、数据库运行时文件、脚本工具配置）          |
| revert   | 撤销某次提交                                                     |
| ci       | CI/CD 流程变更                                                 |

**type 选型优先级**：当一次提交同时涉及多种类型时，按以下优先级选择主 type：

```
fix > feat > refactor > perf > docs > build > chore
```

---

## 三、作用域（scope）词汇表

scope 使用**中文**，从以下列表中选择最贴近的一个；如变更跨越两个及以上模块，用 `&` 连接所有涉及的模块名（如 `文档&数据层`），**禁止**使用笼统的「多模块」标签：

| scope  | 对应目录 / 含义                              |
| ------ | --------------------------------------- |
| 界面     | `ui/`                                   |
| 核心逻辑   | `core/`                                 |
| 服务层    | `services/`                             |
| 数据层    | `data/`（含数据库文件 `.db`）                  |
| 配置     | `config/`                               |
| 工具函数   | `utils/`                                |
| 构建     | `build/`                                |
| 测试     | `test/`                                 |
| 文档     | `help_document/`、`README.md` 等说明文件     |

**多模块 scope 示例**：`docs(文档&数据层)`、`feat(核心逻辑&界面&服务层)`

---

## 四、body 书写规则

1. 按**模块目录**分组，组标题格式为 `### 模块名`（与 scope 词汇表中的中文名对应）
2. 每个文件一条 bullet：`` - `路径/文件名`：[做了什么] → [影响是什么] ``
3. "做了什么"使用具体动词描述操作；"影响是什么"描述对功能、性能、接口或数据的影响
4. 若同模块内只有一个文件变动，组标题可省略，直接写 bullet
5. 纯格式调整（空行、注释措辞、缩进）等琐碎改动可合并为一条，注明"统一格式"
6. 多个同类文件的批量操作（如批量重命名资源）可合并为一条，注明文件数量

---

## 五、特殊文件处理规范

### 数据库文件（`.db`）

- 归入 `数据层` 分组，type 随主变更；若本次提交仅有数据库变更，使用 `chore(数据层)`
- diff 中仅显示 Git LFS 指针（`oid sha256:...`）变化时，描述格式为：
  ```
  - `data/sourceData.db`：LFS 指针更新 → 同步最新运行时数据
  ```
- 若能从上下文推断数据库变更原因（如新增会话、表结构迁移），应在描述中补充说明

### 自动生成文件

- 由工具生成的文件（如 Qt Designer 输出的 `ui_main_window.py`）描述中需注明"（自动生成）"
- 描述重点放在**触发生成的原因**（如"重新生成 UI 文件以应用新增控件"），而非生成内容本身

### 二进制资源文件

- 图片、字体等二进制文件使用"新增 / 替换 / 删除"描述，并说明文件用途
- 不描述二进制文件的具体内容

### 配置运行时文件（`config.txt`）

- **禁止**在提交信息中写入实际的设备地址、密钥等敏感值
- 描述格式：
  ```
  - `config/config.txt`：更新 `TARGET_ADDRESS` 配置项 → 指向新测试设备
  ```

---

## 六、footer 书写规则

- **破坏性变更**：`BREAKING CHANGE: <描述接口/行为变化，以及迁移方式>`
- **关联 issue**：`Closes #<编号>` 或 `Refs #<编号>`（多个用换行分隔）
- 无破坏性变更且无关联 issue 时，**不写 footer**

## 七、完整示例

### 示例 1 — 普通功能新增

```
feat(核心逻辑): 新增蓝牙断线自动重连机制

### 核心逻辑
- `core/managers/bluetooth_manager.py`：新增 `_reconnect_loop` 协程，在连接断开后按指数退避策略重试 → 解决设备短暂离线后主动重连失败的问题
- `core/bluetooth_protocol.py`：新增 `on_disconnect` 回调注册接口，供管理器监听断线事件 → 解耦协议层与重连逻辑

### 服务层
- `services/bluetooth_service.py`：在服务启动时将 `bluetooth_manager` 的重连参数绑定到配置项 `ble.reconnect_interval` → 使重连间隔可通过配置文件动态调整
```

---

### 示例 2 — 含破坏性变更的重构

```
refactor(数据层): 将数据存储接口迁移至异步模式

### 数据层
- `data/database.py`：将所有数据库读写方法改为 `async def`，移除同步阻塞调用 → 消除 UI 线程卡顿，提升整体响应速度
- `data/data_models.py`：新增 `AsyncSession` 类型注解，移除废弃的 `SyncSession` → 统一会话类型，减少类型混用错误

### 核心逻辑
- `core/data_pipeline.py`：使用 `await` 调用更新后的存储接口，新增错误回调处理 → 确保管道在异步环境下正常运行

BREAKING CHANGE: 所有调用 data/database.py 中存储接口的代码需改为异步调用（await），同步调用方式已移除
Closes #42
```

---

### 示例 3 — 含数据库文件的多模块提交

```
docs(文档&数据层): 新增问题追踪文档并修正两份文档的目录结构

### 文档
- `help_document/issues_and_optimizations.md`：新建问题与优化追踪文档，包含 Bug、优化、TODO、已完成归档四个版块，迁移旧备注中的 3 条优化记录（OPT-001 ～ OPT-003）→ 建立规范化的问题追踪机制，替代原无格式备注文件
- `help_document/project_development_outline.md`：完整重写顶部目录树，补充 `build/`、`config/`、`version/` 目录，移除已不存在的 `services/sql_service.py`；末尾新增"相关文档"导航表格 → 使目录结构与当前代码库保持一致
- `README.md`：修正目录树，根目录 `config.py`/`config_loader.py` 移入 `config/` 子目录，补全 `ui/`、`test/`、`build/` 缺失条目；"配置说明"章节新增问题追踪文档链接；更新日志新增 v2.2 条目 → 确保文档与实际项目结构一致

### 数据层
- `data/sourceData.db`：LFS 指针更新 → 同步最新运行时数据
```

## 八、注意事项

- **禁止使用** `update`、`modify`、`change` 等模糊动词，应使用具体动词：新增、修复、移除、重构、优化、迁移、统一、解耦、重写、补全等
- **header 禁止**以句号结尾
- body 中的文件路径使用**项目根目录为基准的相对路径**
- 单文件提交：body 只写一条 bullet，无需分组
- 提交信息全程使用**中文**（type、scope 中的英文关键字除外）
- **不推测**：若 diff 中某文件改动原因不明，如实描述 diff 内容，不编造业务理由
